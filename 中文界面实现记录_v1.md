# GreenMind 中文界面实现记录 v1

## 1. 目的

本记录用于追踪 GreenMind 从英文 TFT 界面迁移到中文界面的每个阶段，记录修改文件、构建结果、资源占用、实机现象和回退方法。若某个阶段在实机上失败，应先根据本记录定位并回退到最近一个已验证阶段，避免同时修改字库、渲染、布局和协议后无法判断原因。

## 3. 分阶段方案

### 阶段 A：只接入字体资源

目标：

- 生成 Unicode→GB2312 完整映射；
- 使用汇编 `INCBIN` 将字体和映射放入 Flash；
- 不修改现有 UI；
- 先确认 Keil 能正常汇编、链接且 Flash 未溢出。

回退：

- 从 Keil 工程移除 `font_assets.s`；
- 删除生成的 Unicode 映射二进制；
- 不影响原英文 UI。

### 阶段 B：加入独立字体服务

目标：

- 支持 UTF-8 解码；
- 二分查找 Unicode→GB2312；
- 支持 12×12 和 24×24 显示；
- 不使用动态内存；
- 字库只驻留 Flash。

回退：

- 从工程移除 `font_service.c` 和 `font_assets.s`；
- 原 `tftlcd.c` 不修改，英文显示路径保持原样。

### 阶段 C：最小中文冒烟测试

目标：

- 先只替换一个标题或状态；
- 烧录后检查是否出现乱码、镜像、越界或白屏；
- 通过后再迁移其他页面。

回退：

- 将 `APP_UI_ENABLE_CHINESE` 设为 `0`，恢复英文绘制路径。

### 阶段 D：逐页中文化

顺序：

1. 底部导航和 SYSTEM；
2. HOME；
3. DETAIL；
4. AI；
5. PLANT LIBRARY；
6. PLANT PROFILE。

每完成一页都重新编译，实机测试时逐页拍照。

## 4. 实现结果

### 2026-07-17：资源和字体服务文件已创建

新增：

- `tools/fonts/generate_unicode_gb2312_map.py`
- `tools/fonts/ui_strings_zh.json`
- `tools/fonts/generate_ui_strings.py`
- `firmware/APP/agriculture/font_assets.s`
- `firmware/APP/agriculture/font_service.c`
- `firmware/APP/agriculture/font_service.h`

设计决定：

- 不把 147 KB 字库复制到 RAM；
- 使用汇编 `INCBIN` 直接链接二进制；
- 使用完整 7445 项 Unicode→GB2312 映射，为以后接收 AI UTF-8 中文文本保留能力；
- ARMCC 源文件不直接包含中文字符串，界面中文统一由生成器转为 UTF-8 数字字节数组；
- 保留英文 UI 路径，中文可以通过配置宏关闭。

### 2026-07-17：阶段 A/B 验证完成

生成结果：

- Unicode→GB2312 映射：7445 项、29780 字节；
- 映射 SHA-256：
  `43600c1fa2ea588467d13207ee970856b660afc36daea279fcf7341983aee716`；
- 字体与映射通过 ARM Compiler 5 的 `armasm INCBIN` 单独汇编；
- `font_assets.s`、`font_service.c` 已加入 `GreenMind.uvprojx`；
- 字体服务能够检查链接资源长度，资源错误时自动回退英文 UI。

只接入资源和字体服务后的构建：

```text
0 Error(s), 1 Warning(s)
```

该次唯一警告是中文就绪变量尚未被 UI 使用；后续页面接入后已消失。

### 2026-07-17：阶段 C/D 软件实现完成

中文覆盖：

保留英文的内容：

- `GreenMind` 品牌名；

- 未知底层错误码和未知 ESP07 状态，便于调试；

- 界面实际使用的非 ASCII 字符：200 个；
- 所有字符均能映射到 GB2312；
- 所有对应字模均非空；
- 字模软件预览已检查，无镜像和行列转置。

### 2026-07-17：英文回退构建已验证

临时把：

```c
#define APP_UI_ENABLE_CHINESE 0
```

后，英文 UI 路径编译通过：

```text
Program Size: Code=47864 RO-data=211688 RW-data=240 ZI-data=14552
0 Error(s), 0 Warning(s)
```


`font_service.c` 和 `font_assets.s`，并恢复 `ui_service.c` 到基线提交。

## 5. 实机验证状态

目前完成的是：

- 字库、映射和文案完整性验证；
- ARM 汇编、链接和固件生成；
- 中英两条 UI 路径编译验证；
- 字模软件预览；
- 网关回归测试。

**中文固件尚未在本次修改后由用户烧录到真实开发板，因此还不能声称六个页面
已经完成实机验收。**

建议第一次烧录后按以下顺序检查：

1. HOME 是否正常显示“环境良好”等中文，且植物图不受影响；
2. KEY_UP 依次检查环境、智能、系统三个主页面；
3. SYSTEM 检查网络状态、IP 和英文底层错误是否仍可读；
4. 点击植物卡片检查植物库和植物档案；
5. 发起 Mock AI，检查发送、思考、校验、完成等中文状态；
6. 观察状态局部刷新时是否残留旧字或出现明显闪烁；
7. 拍摄每页照片并记录问题坐标和文字。

若出现白屏或完全无法启动，优先把 `APP_UI_ENABLE_CHINESE` 设为 `0` 再烧录，
用于区分问题来自中文绘制还是其他模块。若英文正常而中文异常，再检查启动串口：

```text
[FONT] asset=READY ui=CHINESE
```

若显示 `asset=INVALID`，说明链接的字库或映射长度不正确，界面会自动使用英文。
