# 基于大模型的智慧植物养护建议系统

## 最小可行产品（MVP）需求文档

## 1. 项目概述

### 1.1 项目名称

基于大模型的智慧植物养护建议与辅助浇水系统。

### 1.2 项目目标

本项目基于 STM32、温湿度传感器、光敏传感器、Wi-Fi 模块、显示屏、按键、蜂鸣器、5×5 LED 彩灯屏和水泵，实现一个小型智慧植物养护系统。

系统能够采集植物周围的温度、空气湿度和光照数据，将数据通过 Wi-Fi 上传至电脑端后端服务。电脑端调用大模型，对当前环境进行综合分析，并生成简短、可解释的植物养护建议。

大模型仅提供建议，不直接控制水泵。用户阅读建议后，可通过按键手动启动水泵。

### 1.3 MVP 核心目标

MVP 只验证以下完整链路：

1. STM32 采集温度、湿度和光照数据。
2. STM32 连接电脑热点。
3. STM32 将数据发送到电脑后端。
4. 电脑后端调用大模型。
5. 大模型返回养护建议。
6. STM32 接收并显示建议。
7. 用户通过按键手动控制水泵。
8. 系统对高温、传感器故障等情况进行本地提示。

---

## 2. 系统定位

本系统不是全自动温室控制系统，而是一个：

> 基于环境感知和大模型分析的植物养护建议与辅助浇水系统。

系统中的职责划分如下：

* STM32负责传感器采集、本地显示、按键输入、蜂鸣器提示和水泵控制。
* 电脑后端负责接收数据、调用大模型、整理返回结果。
* 大模型负责综合解释环境状态并生成养护建议。
* 用户负责决定是否执行浇水操作。

## 3. 系统总体结构

系统采用以下网络结构：

```text
温湿度传感器 ─┐
光敏传感器   ─┼─> STM32
按键         ─┘      │
                      │ Wi-Fi
                      ▼
              电脑热点（AP）
                      │
                      ▼
             电脑后端服务
                      │
                      ▼
                大模型 API
                      │
                      ▼
             返回植物养护建议
                      │
                      ▼
                   STM32
             ┌────────┼─────────┐
             ▼        ▼         ▼
           屏幕     5×5 LED    蜂鸣器
                                 │
                                 ▼
                           按键控制水泵
```

### 3.1 网络角色

* 电脑：开启 Wi-Fi 热点，作为 AP。
* STM32：通过 Wi-Fi 模块连接电脑热点，作为 STA。
* 后端程序：运行在电脑本机。
* STM32：通过电脑热点访问电脑后端的局域网 IP 地址。

例如：

```text
电脑热点 IP：192.168.137.1
后端端口：8000
接口地址：
http://192.168.137.1:8000/api/analyze
```

实际 IP 地址以电脑热点分配结果为准。

## 5. 功能需求

## 5.1 传感器采集

### FR-01 温度采集

系统应定时读取当前环境温度。

推荐采集周期：

```text
每 2 秒采集一次
```

系统应保存当前温度值，用于：

* 屏幕显示；
* 本地状态判断；
* 上传给电脑后端；
* 大模型环境分析。

### FR-02 空气湿度采集

系统应定时读取当前空气湿度。

系统应保存当前湿度值，用于：

* 屏幕显示；
* 本地状态判断；
* 上传给电脑后端；
* 大模型环境分析。

### FR-03 光照采集

系统应读取光敏传感器数值，并将原始值转换为以下状态之一：

```text
dark：光照不足
normal：光照正常
strong：光照较强
```

MVP 阶段可以根据实验环境自行设定阈值。

例如：

```text
光敏值 < 1000：光照较弱
1000～2500：光照正常
> 2500：光照较强
```

具体阈值需根据传感器实际输出校准。

### FR-04 传感器异常检测

当温湿度传感器或光敏传感器读取失败时，系统应：

* 在屏幕显示传感器异常；
* 将系统状态设置为故障；
* 可选地触发蜂鸣器；
* 不将无效数据发送给大模型。

## 5.2 本地显示

### FR-05 实时数据显示

屏幕应显示：

```text
温度
空气湿度
光照状态
Wi-Fi连接状态
系统状态
```

示例：

```text
T: 28.6 C
H: 52 %
L: Normal
WiFi: OK
```

### FR-06 AI 建议显示

收到电脑后端返回的建议后，屏幕应显示：

* 综合状态；
* 主要问题；
* 简短建议；
* 是否建议检查土壤。

示例：

```text
AI: Warning

Temp is high.
Avoid direct light.
Check soil before watering.
```

由于 STM32 屏幕尺寸有限，后端返回的建议应尽量简短。

### FR-07 页面切换

系统至少提供两个页面：

1. 实时数据页面；
2. AI 建议页面。

用户可以通过按键切换页面。

## 5.3 AI 分析

### FR-08 用户主动请求 AI 分析

用户按下“AI 分析键”后，STM32 应：

1. 获取最新温度、湿度和光照数据；
2. 检查 Wi-Fi 是否连接；
3. 将数据发送到电脑后端；
4. 显示“正在分析”；
5. 等待后端返回结果；
6. 显示 AI 建议。

MVP 阶段以用户主动调用为主。

暂不要求定时自动调用大模型。

### FR-09 后端接收数据

电脑后端应提供 HTTP 接口：

```text
POST /api/analyze
```

STM32 发送的数据格式示例：

```json
{
  "temperature": 31.2,
  "humidity": 39,
  "light_value": 2780,
  "light_level": "strong"
}
```

### FR-10 后端调用大模型

后端收到数据后，应构造模型请求。

模型需要完成以下任务：

* 判断当前环境整体状态；
* 判断主要问题；
* 给出一到三条简短建议；
* 给出是否需要检查土壤的建议；
* 不直接要求启动水泵。

### FR-11 AI 返回结构

大模型返回结果应由后端整理为固定 JSON 格式：

```json
{
  "status": "warning",
  "main_issue": "high_temperature",
  "suggestion": "温度偏高且光照较强，建议避免阳光直射，并检查土壤干湿。",
  "watering_advice": "check_soil"
}
```

允许的 `status`：

```text
normal
warning
danger
error
```

允许的 `watering_advice`：

```text
no_need
check_soil
water_if_dry
unknown
```

### FR-12 后端结果精简

后端应限制返回文本长度，避免 STM32 屏幕无法显示。

建议：

```text
建议文本不超过 80 个中文字符
```

## 5.4 水泵控制

### FR-13 手动浇水

用户长按浇水按键后，系统应启动水泵。

水泵运行固定时间后自动停止。

建议初始配置：

```text
单次运行时间：3 秒
```

### FR-14 水泵安全限制

系统必须具备以下限制：

* 水泵不得无限运行；
* 水泵启动后必须自动停止；
* 水泵运行期间再次按键不得重复启动；
* 传感器故障不影响用户手动停止水泵；
* Wi-Fi 断开时仍可手动控制水泵。

### FR-15 AI 不直接控制水泵

大模型只允许返回浇水建议，不允许直接返回并执行：

```text
pump_on
pump_seconds
start_watering
```

水泵只能由用户按键启动。

MVP 可以使用以下初始规则：

```text
温度 < 10℃：
低温警告

10℃ ≤ 温度 ≤ 30℃：
温度正常

30℃ < 温度 ≤ 35℃：
温度偏高

温度 > 35℃：
严重高温
```

```text
湿度 < 35%：
空气偏干

35% ≤ 湿度 ≤ 75%：
湿度正常

湿度 > 75%：
空气偏湿
```

```text
光照值低于下限：
光照不足

光照值处于正常范围：
光照正常

光照值高于上限：
光照较强
```

这些阈值仅用于 MVP 演示，后续可以根据植物种类调整。

## 7. AI 提示词要求

后端向大模型发送的提示词应明确以下要求：

```text
你是一个植物环境养护助手。

请根据当前温度、空气湿度和光照状态，
分析当前植物环境。

要求：
1. 只给出简短、明确的养护建议。
2. 不得直接控制设备。
3. 不得仅根据空气湿度判断土壤缺水。
4. 涉及浇水时，优先建议用户先检查土壤。
5. 建议文本不超过80个中文字符。
6. 返回固定JSON格式。
```

输入数据示例：

```json
{
  "temperature": 31.2,
  "humidity": 39,
  "light_level": "strong"
}
```

期望模型输出：

```json
{
  "status": "warning",
  "main_issue": "high_temperature_and_strong_light",
  "suggestion": "当前温度偏高且光照较强，建议移至散射光区域，并检查土壤干湿。",
  "watering_advice": "check_soil"
}
```

## 8. 电脑端后端需求

### 8.1 开发技术

推荐使用：

```text
Python
FastAPI
Uvicorn
模型 API SDK 或 HTTP 请求库
```

### 8.2 后端主要接口

#### 健康检查接口

```text
GET /health
```

返回：

```json
{
  "status": "ok"
}
```

用于 STM32 测试是否能访问电脑后端。

#### AI 分析接口

```text
POST /api/analyze
```

请求：

```json
{
  "temperature": 28.6,
  "humidity": 52,
  "light_value": 1800,
  "light_level": "normal"
}
```

响应：

```json
{
  "success": true,
  "status": "normal",
  "main_issue": "none",
  "suggestion": "当前温湿度和光照基本正常，暂时无需调整。",
  "watering_advice": "no_need"
}
```

### 8.3 API Key 保存

模型 API Key 只能保存在电脑后端。

不得将 API Key：

* 写入 STM32 代码；
* 写入 Wi-Fi 模块代码；
* 上传到公开代码仓库；
* 直接发送给 STM32。

建议使用环境变量：

```text
MODEL_API_KEY=xxxxxxxx
```

## 9. 网络通信需求

### 9.1 热点连接

电脑开启移动热点。

STM32 Wi-Fi 模块设置为 STA 模式，连接以下信息：

```text
SSID：电脑热点名称
Password：电脑热点密码
```

### 9.2 后端监听地址

后端不能只监听：

```text
127.0.0.1
```

应监听：

```text
0.0.0.0
```

例如：

```bash
uvicorn main:app --host 0.0.0.0 --port 8000
```

否则 STM32 无法通过局域网访问电脑。

### 9.3 Windows 防火墙

电脑需要允许后端端口通过防火墙。

例如允许：

```text
TCP 8000端口
```

### 9.4 网络测试顺序

网络联调应按以下顺序进行：

1. 电脑开启热点；
2. STM32 连接热点；
3. 查看 STM32 获得的 IP；
4. 确认电脑热点网卡 IP；
5. 电脑启动后端；
6. 手机连接同一热点测试 `/health`；
7. STM32 请求 `/health`；
8. STM32 请求 `/api/analyze`。

MVP 可以使用三个按键。

| 按键   | 功能              |
| ---- | --------------- |
| KEY1 | 切换实时数据页和 AI 建议页 |


## 11. 主要业务流程

### 11.1 正常监测流程

```text
启动系统
→ 初始化传感器
→ 初始化屏幕
→ 连接电脑热点
→ 定时读取温湿度和光照
→ 更新屏幕
→ 更新LED状态
→ 等待用户操作
```

### 11.2 AI 分析流程

```text
用户按下AI分析键
→ 检查传感器数据是否有效
→ 检查Wi-Fi是否连接
→ 向电脑后端发送HTTP请求
→ 屏幕显示“AI分析中”
→ 接收后端JSON响应
→ 保存AI建议
→ 屏幕显示建议
```

### 11.3 浇水流程

```text
用户长按浇水键
→ 判断水泵是否正在运行
→ 启动水泵
→ 显示“正在浇水”
→ 运行3秒
→ 自动关闭水泵
→ 显示“浇水完成”
```

### 11.4 网络异常流程

```text
AI请求时发现Wi-Fi未连接
→ 不发送请求
→ 屏幕显示“Wi-Fi未连接”
→ 蜂鸣器提示
→ 系统继续本地监测
```

## 12. 非功能需求

### 12.1 实时性

* 传感器数据每 2 秒更新一次；
* 按键响应时间不超过 200 毫秒；
* 水泵关闭必须由本地定时器控制；
* AI 响应建议在 15 秒内返回。

### 12.2 可用性

即使出现以下情况，系统仍应保持基本功能：

* Wi-Fi 断开；
* 模型 API 请求失败；
* 模型返回格式错误；
* 电脑后端未启动。

网络不可用时，系统仍应：

* 采集传感器数据；
* 显示实时状态；
* 执行本地报警；
* 允许用户手动控制水泵。

### 12.3 安全性

* API Key 不得保存在 STM32；
* 水泵必须限时运行；
* AI 不得直接控制水泵；
* 异常数据不得作为有效分析输入；
* 模型响应需要经过后端校验。

## 14. 验收标准

### AC-01 传感器采集

系统能够正确显示：

* 温度；
* 空气湿度；
* 光照状态。

### AC-02 Wi-Fi连接

STM32 能够以 STA 模式连接电脑热点。

### AC-03 后端访问

STM32 能够成功请求电脑端：

```text
GET /health
```

并接收正常响应。

### AC-04 AI分析

用户按下 AI 分析键后，系统能够：

* 上传传感器数据；
* 获得大模型建议；
* 在屏幕上显示建议。

### AC-05 异常处理

关闭电脑后端或断开热点时：

* 系统不死机；
* 屏幕提示网络异常；
* 传感器采集仍正常运行。

### AC-06 水泵控制

用户长按水泵按键后：

* 水泵启动；
* 运行固定时间；
* 自动停止；
* 不会持续运行。

### AC-07 AI权限限制

大模型返回结果不会直接启动水泵。

### AC-08 状态提示

环境异常时，屏幕、LED 或蜂鸣器至少有一种明显提示。

## 15. 推荐开发顺序

### 第一阶段：电脑端模拟测试

暂时不连接 STM32。

使用浏览器、Postman 或 Python 脚本向后端发送模拟数据：

```json
{
  "temperature": 33,
  "humidity": 36,
  "light_value": 3000,
  "light_level": "strong"
}
```

确认后端能够：

* 接收 JSON；
* 调用模型；
* 返回固定格式建议。

### 第二阶段：STM32基础采集

实现：

* 温湿度读取；
* 光敏传感器读取；
* 屏幕显示；
* 按键输入；
* 水泵运行和自动停止。

## 17. MVP核心总结

本系统的最小闭环为：

```text
环境感知
→ 本地显示
→ 用户请求AI分析
→ 大模型生成养护建议
→ 用户判断
→ 手动启动水泵
```

项目的核心创新点为：

1. 利用大模型综合解释温度、湿度和光照数据；
2. 将传统阈值报警升级为自然语言养护建议；
3. 大模型只提供建议，不直接控制执行器；
4. 系统在网络和模型不可用时仍能独立运行；
5. 采用电脑热点和本地后端，降低初期开发与测试难度。
