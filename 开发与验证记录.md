## 已作出的设计决定

### ADR-001：电脑作为热点和模型网关

- 状态：采用
- 原因：电脑同时具备互联网、大模型 SDK、日志和调试能力；避免手机连接 ESP07 后失去网络。
- 结果：电脑为 WiFi AP，ESP07 为 STA，Python 网关运行在电脑。

### ADR-002：设备链路使用原始 TCP，不让 STM32 直接 HTTP

- 状态：采用
- 原因：当前 TCP Server 已实物验证；STM32 直接处理 HTTP、HTTPS 和大 JSON 会增加 RAM、解析和 AT 状态复杂度。
- 结果：Python 网关连接 ESP STA IP:8080，并代替 STM32 调用模型 HTTPS API。

### ADR-003：模型只返回建议，不控制水泵

- 状态：采用
- 原因：执行器必须具备确定性和本地安全边界。
- 结果：正式 TCP 协议没有水泵启动命令，水泵只由本地长按按键启动。

### ADR-004：TFT 第一版不显示任意动态中文

- 状态：采用
- 原因：现有 TFT 驱动只有少量固定中文字模，不能显示模型返回的任意中文。
- 结果：STM32 显示状态码、建议码对应的 ASCII/固定英文短句；完整英文在电脑端显示。

### ADR-005：独立建立 AgricultureSys 综合工程

- 状态：采用
- 原因：避免破坏 `wifi_tft` 和各模块实验的已验证状态。
- 结果：后续从基线复制到 `AgricultureSys/firmware`，原实验只作参考和回退。

### 2026-07-17：修复 AI_RESULT 突发消息丢失

- 复现条件：Mock 网关在同一秒连续发送 `ACK`、三个 `AI_STAGE` 和 `AI_RESULT`；TFT 停留在 `VALIDATING`，45 秒后显示 `MODEL TIMEOUT`。
- 根因：原协议环形队列配置为 4 个槽位，实际容量只有 3 条事件；队列满时静默丢弃新事件，最终 `AI_RESULT` 可能被丢弃。
- 修改文件：`APP/agriculture/app_config.h`、`protocol.c/.h`、`User/main.c`。
- 修复：队列改为 8 个槽位（实际容量 7 条），足以容纳当前 5 条突发消息。
- 异常处理：增加协议事件丢弃计数；若仍溢出，TFT 显示 `EVENT OVERFLOW`，USART1 输出 `[PROTOCOL] EVENT OVERFLOW count=...`，AI 请求立即进入错误状态而非等待到超时。
- 编译：0 errors / 0 warnings。
- 输出文件：`firmware/Obj/Template.hex`，生成时间 2026-07-17 09:44:50。
- 待实物验证：继续使用 `--provider mock`，确认界面从 `VALIDATING` 进入 `DONE` 并显示 `NORMAL / NONE / NO_NEED / KEEP_CURRENT`。
