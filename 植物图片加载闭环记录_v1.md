### 2. 在线植物按需加载

`CACTUS` 用作第一条实物测试链路，但网关接口本身支持数据库中的全部植物和三种状态。

```text
选择 CACTUS
→ SQLite 更新 current_species_id
→ STM32 收到 PLANT_SELECTED
→ 请求 cactus/NORMAL
→ 网关查询 plant_images
→ 读取 cactus_normal.rgb565
→ 分块发送
→ STM32 重组并计算 CRC32
→ TFT 显示仙人掌
```

图片为 `48×48 RGB565_BE`，每张 4608 字节。TFT 主页使用 3 倍显示，最终占用 `144×144` 像素区域。绘图按相同行和水平色块合并，避免逐像素设置 LCD 窗口。

## 图片协议

STM32 请求：

```text
V1|ASSET_REQ|TRANSFER_ID|SPECIES_ID|STATE
```

示例：

```text
V1|ASSET_REQ|7|cactus|NORMAL
```

网关发送元数据：

```text
V1|ASSET_BEGIN|7|cactus|NORMAL|48|48|4608|116|388CB035
```

随后发送 116 个十六进制分块，每块最多 40 字节原始数据：

```text
V1|ASSET_CHUNK|7|0|<HEX>
V1|ASSET_CHUNK|7|1|<HEX>
...
```

结束帧：

```text
V1|ASSET_END|7|116|388CB035
```

失败帧：

```text
V1|ASSET_ERROR|7|ASSET_NOT_FOUND
```

网关每 20 ms 发送一个分块，以适配固件当前 256 字节 TCP 流接收缓冲区。分块日志被压缩为每 32 块显示一次进度，避免终端被 116 行 HEX 内容刷屏。

## 固件异常处理

- 尺寸必须为 `48×48`。
- 总长度必须为 4608 字节。
- 分块编号必须从 0 连续递增。
- 十六进制字符、分块长度、像素数量都要校验。
- `ASSET_END` 的分块数和 CRC32 必须与 `ASSET_BEGIN` 一致。
- CRC、分块、超时或网关错误时自动重试一次。
- 新植物请求开始后，旧传输 ID 的剩余分块会被忽略。
- 8 秒未完成则进入 `ASSET TIMEOUT`。
- 最终失败时主页显示通用植物和 `IMG ERROR`，不会把绿萝冒充为其他植物。
- 加载中显示 `LOADING`。
- TCP 断开时终止正在进行的图片传输，但不阻塞传感器和触屏任务。

## 2026-07-17 卡死修复

实物验证发现：网关连接后，屏幕和触摸会永久无响应。根因是图片协议扩大了
`ProtocolEvent`，但启动文件仍只分配 1 KB 栈。连接后的调用链同时在栈中保存
`AppState`、TCP 事件和协议解析事件，实际栈占用超过原有上限，导致内存被覆盖。

修正内容：

- `AppState`、TCP 事件和协议解析临时事件改为静态存储，避免大对象反复压栈。
- 工程栈从 1 KB 增加至 2 KB，作为后续扩展的安全余量。
- 首阶段动态图片传输严格限制为 `CACTUS`；`POTHOS` 使用内置图片，其他植物暂用兜底图。
- 这样数据库当前植物即使是 `ORCHID`，网关连接时也不会自动开始大批量传图。

反汇编复核的主要函数局部栈已经从修复前的大约 336/324 字节下降为：

```text
main                 32 bytes
AppTcp_Task          64 bytes
Protocol_ParseFrame  52 bytes
Protocol_BadFrame     0 bytes
```

修复后重新执行 29 项 Python 测试并全量编译，结果为
`0 Error(s), 7 Warning(s)`。
